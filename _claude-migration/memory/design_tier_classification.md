# Tier classification — the "ALBEDO genius" reframe (decided 2026-06-16, build NEXT SESSION)

This is the agreed design direction for **sub-task 2** of the A3 production wiring (the last remaining one — see [[project_b_hooks_and_head]]). User explicitly chose to replace the weak name-heuristic with a dataflow-based design and start implementing **step 1 next session**.

## The problem (grounded in real code)
- `is_interactive` is set from a **component-NAME heuristic** at `src/estimator.rs:79` (`estimate_priority_hints`: name contains button/form/input/link). The scanner copies it onto the component at `src/scanner.rs:110` (`component.is_interactive = hints.is_interactive`). The node binding does the same at `crates/albedo-node/src/lib.rs:348`.
- That flag is THE lever: `src/effects.rs:79 decide_tier_and_hydration(effects, is_interactive, ...)` routes a **hooks** component to **Tier-C (client island, zero round-trip) iff `is_interactive`, else Tier-B (server round-trip per interaction)**. (It also already sends `io`→Tier-C and `side_effects`→Tier-C unconditionally; `async`→C if interactive/heavy else B.)
- So the name heuristic decides client-hydration-vs-server-round-trip. It errs BOTH ways: false negatives (`Counter`, `Toggle`, `Dropdown` with onClick+useState wrongly land Tier-B — this is exactly why CounterButton had to be renamed in `A:\_albedo_verify`), false positives (`LinkList`/`FormStatus` promoted with no real interactivity).

## The reframe (the actual decision)
Tier is **a provable property of dataflow, not a label.** The real question separating Tier-B from Tier-C is: **"Can this component's state transitions be computed entirely client-side, or do they need the server?"** A compiler with a component graph + effect analysis can *prove* this instead of guessing — the differentiator a bolt-on framework (React/Next, which forces manual `"use client"`) structurally cannot do.

Derivation per event handler:
- Take each `on*` handler's closure body.
- Collect free variables + calls; classify each: local `setState`/`useState` value / pure helper → **client-satisfiable**; `fetch` / `action()` / server-only import / secret / DB → **server boundary**.
- Handler closure ALL client-satisfiable → **Tier-C** (hydrate, zero round-trip). Any server touch → **Tier-B**. Can't prove → **Tier-B** (safe).
- This distinguishes a local `Counter` (Tier-C) from a `LikeButton` whose onClick `await fetch('/api/like')` (Tier-B) — both have onClick; handler-PRESENCE alone can't tell them apart, dataflow can.

## Soundness direction (makes it principled, not reckless)
**prove-client-safe → C; prove-needs-server → B; can't-prove → B (round down).** A wrong Tier-C *breaks* (client can't reach the data); a wrong Tier-B is merely *slower*. Always round down on uncertainty. `"use client"` / `"use server"` string-literal directives survive as the explicit override for the long tail the analysis can't crack (cross-module helpers, dynamic dispatch, indirect calls — conservatively treated as server until proven pure). Matches the RSC mental model React devs already know.

## Second layer (later): sub-component island extraction
Once dataflow exists, don't tier the whole component — tier the **subtree**. Identify the minimal JSX subtree that depends on mutating state, ship ONLY that as the island, keep the rest Tier-A. "Islands within components," automatic, no `<Suspense>`/boundary authoring. Makes the "ships almost no JS" pitch real; only a compiler can do it.

## Why it's ~70% built already
`src/parser.rs:230 EffectCollector` is an SWC `Visit` walker already classifying `hooks`/`io`/`async`/`side_effects` (per `mark_call` → `is_hook_call` etc.). It does NOT yet look at JSX attributes. `decide_tier_and_hydration` already graph-routes io/side_effects to Tier-C. The component graph has dependency edges. The work is to **scope the effect analysis to per-handler closures + resolve free variables**, not invent an analysis.
